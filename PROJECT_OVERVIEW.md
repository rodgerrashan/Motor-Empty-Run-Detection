# Motor Empty Run Detection - Full Project Explanation

## Purpose

This repository is a runnable, full-stack MVP for detecting motor "empty run" (and other abnormal states) from live telemetry. It demonstrates an end-to-end industrial telemetry flow:

Sensors or simulators -> MQTT -> real-time inference -> storage -> live dashboard

## What "Empty Run" Means

In this project, "empty run" means the motor is running without load (for example, a disconnected belt). In simulation, it appears as:

- Normal-ish RPM
- Lower current draw
- Different vibration patterns
- Lower power factor

The backend predicts states such as `NORMAL`, `EMPTY_RUN`, `STALLED`, and `OFF`.

## High-Level Runtime Flow

1. A sensor simulator publishes telemetry as JSON to MQTT.
2. Mosquitto broker forwards messages to subscribers.
3. The backend ingests telemetry, engineers features, runs ONNX inference, and stores results in MongoDB.
4. Alerts and evaluation snapshots are created when needed.
5. The frontend pulls initial history via REST and listens for live updates via WebSocket.

## Data and API Details

### REST API

- `GET /health` - Health check
- `GET /api/telemetry/latest?limit=30&motorId=MTR-001` - Latest telemetry
- `GET /api/alerts/recent?limit=20` - Recent alerts
- `GET /api/eval/latest` - Latest model evaluation snapshot

### WebSocket Messages

WebSocket endpoint: `ws://<backend-host>:8080/ws`

Message envelope:

```json
{
  "type": "telemetry" | "alert" | "model_eval" | "connected",
  "payload": {},
  "ts": 1710000000000
}
```

### MongoDB Collections

- `telemetry` - One document per telemetry event
- `alerts` - Abnormal events (empty-run, stalled, off, or alert codes)
- `model_eval` - Rolling-window model health summaries

## Feature Engineering Summary

Runtime features align with the training notebook. Examples include:

- `vib_per_amp`
- `current_ratio`
- `vibration_dev`
- `temp_rise`
- `power_kw`
- `high_current_low_rpm`

The model input order is defined by [mlModel/feature_columns.json](mlModel/feature_columns.json).

## How to Run

### Docker Compose (recommended)

```bash
docker compose up --build
```

Services:

- Frontend: http://localhost:5173
- Backend API: http://localhost:8080
- Backend health: http://localhost:8080/health
- MongoDB: localhost:27017
- MQTT broker: localhost:1883

### Local Sensor Publisher (optional)

If you stop the containerized sensor publisher, you can run:

```bash
python sensors/mqtt_publisher.py
```

Supported environment variables:

- `MQTT_BROKER`
- `MQTT_PORT`
- `MQTT_TOPIC`
- `PUBLISH_INTERVAL_SECONDS`

## What Happens During a Single Telemetry Event

1. Publisher sends telemetry to MQTT.
2. Backend receives telemetry, validates it, and computes engineered features.
3. ONNX model infers the operating state.
4. Telemetry is saved to MongoDB.
5. Alerts are created if the state is abnormal or the publisher flagged an alert.
6. Periodic evaluation snapshots are stored.
7. Frontend receives live updates over WebSocket.
